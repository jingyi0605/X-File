# X-File 大目录压测说明

这份说明记录 `scripts/benchmark-large-library.mjs` 的用法。脚本会生成临时资料库目录和大量文件，直接调用 `@x-file/indexer` 的 `runLibraryIndexOnce(...)`，再读取 `.ai-index/exports/manifest.json` 和导出产物，最后输出 JSON 摘要。

## 默认小规模验证

默认参数用于日常冒烟，不应该拖慢本地开发：

```bash
pnpm benchmark:large-library
```

等价于：

```bash
pnpm benchmark:large-library -- --files 500 --dirs 25 --bytes 2048 --extension .md
```

默认会创建系统临时目录，索引结束后删除。如果要复查 `.ai-index`，加 `--keep`：

```bash
pnpm benchmark:large-library -- --keep
```

## 大规模实机参数

大规模压测会占用明显磁盘、CPU 和内存，只适合在实机空闲时跑。建议先用 1 万文件确认环境，再跑 10 万文件：

```bash
pnpm benchmark:large-library -- --files 10000 --dirs 500 --bytes 4096 --keep --out docs/large-library-10k-result.json
```

```bash
pnpm benchmark:large-library -- --files 100000 --dirs 1000 --bytes 4096 --keep --out docs/large-library-100k-result.json
```

如果要压测指定目录，可以传 `--root`。注意脚本会向该目录写入大量文件，不要指向真实用户资料库：

```bash
pnpm benchmark:large-library -- --root /tmp/x-file-large-library --files 100000 --dirs 1000 --bytes 4096 --keep
```

## JSON 摘要怎么看

输出重点字段：

- `index.fallbackMode`：是否走了 `better-sqlite3` 不可用时的兜底导出路径。
- `index.scannedCount`：扫描到的文件数量。
- `index.indexedCount`：索引成功数量。
- `index.failedCount`：失败数量。
- `export.metaDocumentCount`：manifest 里 meta 分片统计的文档数量。
- `export.statusDocumentCount`：`status.json` 里的文档数量。
- `process.rssMb`：脚本结束时进程 RSS 内存。
- `durationMs`：生成文件、索引和读取导出摘要的总耗时。

好结果不是“没有输出错误”就算完。至少要确认 `indexedCount`、`metaDocumentCount` 和预期文件数一致，`failedCount` 为 0。

## Node 25 和 better-sqlite3 fallback

当前 `@x-file/indexer` 正式路径使用 `better-sqlite3`。如果 Node 25 环境没有匹配的 native binding，`runLibraryIndexOnce(...)` 会捕获绑定加载失败，并走 fallback 导出：扫描文件后直接写出可读的静态导出产物。

这不是正式 SQLite 写库路径，只能说明扫描和导出兜底可用。压测报告里必须保留 `index.fallbackMode`：

- `fallbackMode: false`：走 SQLite 正式路径，结果更接近生产行为。
- `fallbackMode: true`：走 Node 25 / native binding 不可用时的兜底路径，只能作为兼容性和导出产物验证。

如果要验证正式路径，使用项目支持的稳定 Node 版本，并确认 `better-sqlite3` 已按当前平台重新安装。

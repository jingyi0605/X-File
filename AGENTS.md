# 项目规则

## 沟通与文档

- 始终使用中文与用户沟通。
- 代码注释、文档、规范使用中文；变量名、函数名、类名等标识符使用英文。
- 新增说明类 Markdown 文档必须使用中文名，并以当前日期开头，例如 `20260609-文档名称.md`。
- 提交信息使用中文。

## 服务器调试

- 不要主动启动前端或后端开发服务器。
- 只有用户明确要求启动并跟踪日志时，才启动开发服务器。

## 前端 UI 设计语言

- 进行任何前端代码编写前，必须先阅读 `docs/20260609-前端开发规范.md`，并按其中的检查清单执行。
- X-File 前端必须复刻并延续 CodingNS 主仓库的全局样式体系。
- 新增页面、元素、组件、弹窗、表单、按钮、标签、空状态和状态提示，必须优先消费 `apps/web/src/styles.css` 中的全局 token 与统一类名。
- 禁止在组件内随手写局部魔法值来覆盖视觉体系；如果确实需要新视觉能力，先扩展全局 token，再让组件消费 token。
- 按钮统一使用 `.primary-button`、`.secondary-button`、`.danger-button`，默认高度由 `--app-button-height` 管理；不要在单个页面把按钮撑成特殊高度。
- 表单控件统一消费 `--control-height-*`、`--modal-control-*`；模态框统一消费 `--modal-*` token 和 `DesktopModal`/`ModalAtoms`。
- 后续前端开发前，必须先检查当前全局样式是否已有对应元素或组件样式；能复用就复用，不能复用再补全全局样式。
- 双栏表单、初始化页、设置页、带插图的卡片布局必须显式规避 CSS Grid/Flex 默认 stretch 导致的控件高度异常。

## 代码质量

- 优先简化数据结构和状态流，避免为特殊情况堆 if/else。
- 保持向后兼容，不破坏现有用户路径。
- 函数短小直接，复杂度超过三层缩进时优先重构。
